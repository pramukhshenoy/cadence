## Findings

### Finding 1 — server/eslint.config.js (HIGH)
**Issue:** Spreading `tseslint.configs.recommended.rules` uses the legacy `.eslintrc` format. In `@typescript-eslint` v8 flat config, the correct key is `tseslint.configs['flat/recommended']` (an array of flat config objects). The old approach manually duplicated parser/plugin setup but skipped the preset's own `languageOptions` and `plugins` registration — future rule additions via the preset would silently not apply.  
**Fix applied:** Replaced `...tseslint.configs.recommended.rules` with `...tseslint.configs['flat/recommended']`, removed redundant manual `plugins` declaration, and added a `files: ['eslint.config.js']` override to exempt the config file from the `no-require-imports` rule triggered by the new preset.

### Finding 2 — server/src/index.ts:6 (LOW)
**Issue:** `process.env.PORT ?? 3000` — `PORT` is `string | undefined` from env; `??` returns `string` when set, keeping the variable as `string | number`. Express accepts both, so no runtime failure. The PORT value `"0"` would not fall back to `3000` (truthy string).  
**Assessment:** Not fixed — no real-world failure case for a dev server with a fixed port. Will be addressed if env validation is added in Phase 1b.

## Verdict: 1 fix applied, 1 deferred
