# Phase 1c — Sonnet Review — 2026-06-23

## Scope

Files reviewed: `server/prisma/schema.prisma`, `server/prisma/migrations/.../migration.sql`, `server/prisma/seed.ts`, `server/package.json`

## Findings

### CONFIRMED

**1. HabitCompletion missing `@@unique([habitId, localDate])`** — `schema.prisma:32`
- Without a DB-level uniqueness constraint, the same habit could be completed twice on the same local date. Streak counts would silently double-count; no guard at the DB or application layer would prevent it.
- **Fix applied**: Added `@@unique([habitId, localDate])` to HabitCompletion. Regenerated migration cleanly.

### PLAUSIBLE

**2. Settings singleton INSERT missing from migration SQL** — `migration.sql:70`
- CLAUDE.md: "Settings singleton row must be seeded in the Prisma migration." The migration created the Settings table but contained no INSERT. `prisma migrate deploy` (production) never runs `prisma db seed`, leaving the row absent with no application-layer fallback.
- **Fix applied**: Added `INSERT OR IGNORE INTO "Settings" ("id", "updatedAt") VALUES ('singleton', CURRENT_TIMESTAMP)` to the migration SQL. Rebuilt the database from this updated migration via `prisma migrate deploy`.

**3. `postinstall: prisma generate` fails on `npm install --omit=dev`**  — `package.json:12`
- `prisma` is in devDependencies. A production install skipping dev deps would fail the postinstall script. Risk is low for this personal project (documented deploy workflow uses plain `npm install`), but is a real failure path.
- **Deferred**: No fix applied. Acceptable for this project's single-machine deploy model.

## Verdict: PASS (fixes applied for findings 1 and 2)
