# Phase 1c — Opus Review — 2026-06-23

## Scope

Files reviewed (post-Sonnet fixes): `server/prisma/schema.prisma`, `server/prisma/migrations/.../migration.sql`, `server/prisma/seed.ts`, `server/package.json`, `server/src/`

## Findings

### CONFIRMED

**1. `seed.ts` excluded from `tsc --noEmit` and ESLint** — `server/prisma/seed.ts:1`
- `tsconfig.json` has `"include": ["src/**/*"]` and `eslint.config.js` has `files: ['src/**/*.ts']`. `prisma/seed.ts` lives outside `src/`, so type errors and lint violations in it are invisible to the quality gate.
- **Fix applied**: Removed `seed.ts` entirely (migration INSERT is the canonical seed; see below).

**2. upsert-on-read safety net absent from `server/src/`** — `server/src/index.ts:1`
- CLAUDE.md: "Settings singleton row must be seeded in the Prisma migration — upsert-on-read as a safety net." The migration INSERT covers the first half. The application-layer safety net — a helper that upserts the row if absent — did not exist anywhere in `server/src/`. Any Phase 2+ route reading Settings without this guard would crash on null if the DB was in a pre-migration state.
- **Fix applied**: Created `server/src/lib/settings.ts` with `getSettings(prisma)` — does a `findUnique` first (no updatedAt bump on the normal path) and only upserts if the row is absent.

**3. `update: {}` in `seed.ts` silently bumps `updatedAt` on every re-run** — `server/prisma/seed.ts:6`
- Prisma's `@updatedAt` injects `SET updatedAt = <now>` into any UPDATE statement, even one with no payload. Every `prisma db seed` re-run against an existing row would touch `updatedAt`, silently misrepresenting "last modified."
- **Fix applied**: Removed `seed.ts` entirely, eliminating both this finding and finding 1. The migration INSERT is the canonical seed; `getSettings()` in `server/src/lib/settings.ts` is the application-layer safety net.

### PLAUSIBLE

**4. `Settings.timezone` ambiguous vs. per-request `X-Timezone` header** — `schema.prisma:86`
- CLAUDE.md: "Backend uses this [X-Timezone header] for all local-time math." `Settings.timezone` is a persisted field with the same bare name as what the middleware provides per-request. A future route author could read the stale stored field instead of the live header, producing wrong reschedule times when the user travels.
- **Fix applied**: Added a disambiguating comment in schema.prisma: `// stored user preference — always use X-Timezone request header for live local-time math`.

### REFUTED

- **`CURRENT_TIMESTAMP` produces Invalid Date**: Empirically refuted — `prisma.settings.findUnique()` returned a valid ISO 8601 string, confirming Prisma 6's library engine correctly parses SQLite's space-separated datetime format.

## Verdict: PASS (all confirmed findings fixed)
