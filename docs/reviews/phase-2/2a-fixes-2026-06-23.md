# Phase 2a — Fixes Summary
**Date:** 2026-06-23  
**Sub-phase:** 2a — Tasks REST API

## Sonnet Fixes Applied

### Fix S1: Invalid `dueDate` string returns 400 instead of 500
**File:** `server/src/routes/tasks.ts` lines 60–67 (POST) and 112–119 (PATCH)  
Before: `new Date(dueDate)` on an invalid string produced `Invalid Date`; Prisma called `.toISOString()` on it, throwing `RangeError` → caught by catch → `next(err)` → 500.  
After: `isNaN(parsed.getTime())` guard returns 400 with a clear error before Prisma is called.  
Tests added: "returns 400 for invalid dueDate string" (POST and PATCH).

### Fix S2: Empty PATCH body returns 400 instead of silently no-oping
**File:** `server/src/routes/tasks.ts` lines 126–129  
Before: `PATCH /api/tasks/:id` with `{}` built an empty `data` object; Prisma no-oped and returned 200 with unchanged data.  
After: `Object.keys(data).length === 0` check returns 400 with "No valid fields provided for update".  
Tests added: "returns 400 for empty body", "returns 400 when body has no recognised fields".

### Fix S3: dotenv loads before PrismaClient construction
**Files:** `server/src/config.ts` (new), `server/src/index.ts` (updated)  
Before: `import app from './app'` in index.ts compiled to `require('./app')` before `dotenv.config()` ran, so any module reading `process.env` at load time would see an empty environment.  
After: `import './config'` as the first import in index.ts ensures dotenv runs before app.ts and its transitive dependencies load.

## Opus Fixes Applied

### Fix O1: PATCH `dueDate: <non-string, non-null>` returns 400 instead of silently clearing
**File:** `server/src/routes/tasks.ts` lines 112–126  
Before: Any non-string `dueDate` value (e.g. `42`, `false`) fell to `else { data.dueDate = null }`, silently wiping the task's due date with a 200 response.  
After: Three-way split — string → parse & validate; `null` → clear (intentional); anything else → 400.  
Tests added: "returns 400 for non-string non-null dueDate", "accepts null dueDate to clear the field".

### Fix O2: `req.body` null guard prevents TypeError on missing Content-Type
**File:** `server/src/routes/tasks.ts` lines 39 (POST) and 87 (PATCH)  
Before: `const { ... } = req.body as { ... }` — if `express.json()` skipped parsing (wrong/missing Content-Type), `req.body` was `undefined`; destructuring threw `TypeError` → caught by catch → 500.  
After: `(req.body ?? {}) as { ... }` — missing body defaults to empty object; the existing field validations then return appropriate 400s.

## Deferred Findings

### Deferred D1: `dueDate` not timezone-adjusted (pre-Phase-3a)
`new Date("2026-06-24")` stores midnight UTC regardless of `req.timezone`. Invisible in Phase 2 (mobile re-applies device timezone at render). Will cause wrong dates in Phase 3a AI system prompt context and Phase 6a notification scheduling. Fix before Phase 3a: require mobile to send full ISO offset strings, or use `req.timezone` + a date library.

### Deferred D2: `GET /api/tasks` missing `?sortBy=` parameter (Phase 2c)
Default sort is `[{ dueDate: 'asc' }, { createdAt: 'desc' }]` with no priority dimension. Phase 2c spec requires "grouped by priority or due date". A `?sortBy=priority|dueDate` query parameter is needed. `dueDate ASC` also puts NULLs last (SQLite default), which may not be the desired UX.
