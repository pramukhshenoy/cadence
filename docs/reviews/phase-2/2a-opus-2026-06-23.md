## Findings

### [CONFIRMED] 1. PATCH `dueDate` non-null non-string silently wipes the field instead of returning 400
**File:** `server/src/routes/tasks.ts` line 121  
**Severity:** High  
**Scenario:** The PATCH `dueDate` block has `else { data.dueDate = null }`. For JSON `null` this is intentional (clear the due date). But for any other non-string value — `42`, `false`, `["2026-01-01"]` — the else branch fires identically, silently setting `dueDate = null`. The task's due date is wiped; the caller receives 200 with no error indication.  
**Fix:** Split the else: `else if (dueDate === null) { data.dueDate = null; } else { res.status(400)... }`

### [PLAUSIBLE] 2. `req.body` destructuring throws TypeError when `Content-Type` is not `application/json`
**File:** `server/src/routes/tasks.ts` lines 39 (POST) and 87 (PATCH)  
**Severity:** Medium  
**Scenario:** `express.json()` does not initialise `req.body` to `{}` when it skips (wrong/missing Content-Type). Express leaves `req.body` as `undefined`. Destructuring `undefined` throws `TypeError: Cannot destructure property 'title' of undefined`. The `catch` block catches it and routes to `next(err)` → errorHandler → 500. A valid authenticated client omitting the Content-Type header gets an opaque 500.  
**Fix:** Guard before destructuring: `const { title, ... } = (req.body ?? {}) as { ... }`

### [PLAUSIBLE] 3. `dueDate` parsed as midnight UTC, ignoring `req.timezone` — will break Phase 3a and 6a
**File:** `server/src/routes/tasks.ts` lines 62 and 114  
**Severity:** Medium (latent — not observable in Phase 2, breaks in Phase 3a and 6a)  
**Scenario:** `new Date("2026-06-24")` stores `2026-06-24T00:00:00.000Z` (midnight UTC). For a UTC+5:30 user, the stored value is 5.5 hours off. The mobile UI re-applies device timezone at render time so it looks correct on screen — but the Phase 3a AI system prompt reads tasks server-side (wrong date shown to AI) and Phase 6a notification scheduling compares UTC timestamps server-side (fires a day early/late).  
**CLAUDE.md rule:** "Backend uses this [X-Timezone header] for all local-time math."  
**Fix (deferred to pre-Phase-3a):** Either (a) require the mobile client to send full ISO offset strings (`"2026-06-24T00:00:00+05:30"`), or (b) use `req.timezone` + a date library to parse date-only strings as local midnight.

### [PLAUSIBLE] 4. `GET /api/tasks` sort omits priority; Phase 2c screen cannot implement "grouped by priority" without a backend change
**File:** `server/src/routes/tasks.ts` line 29  
**Severity:** Low-Medium  
**Scenario:** `orderBy: [{ dueDate: 'asc' }, { createdAt: 'desc' }]` sorts only by due date. The spec (phases.md 2c) requires "Task list screen grouped by priority or due date." The backend provides no priority-sort option. Phase 2c will need a `?sortBy=` parameter added to this endpoint. Also: SQLite `dueDate ASC` puts NULLs last, so undated HIGH-priority tasks sink below dated LOW-priority ones.  
**Fix (deferred to Phase 2c):** Add `?sortBy=priority|dueDate` query parameter. For now noting the gap.

### [REFUTED] 5. Import ordering fragility in index.ts
The failure mode (startup throws immediately) is loud, the codebase is single-developer, and no existing imports in the chain read env vars at load time. Not a real risk.
