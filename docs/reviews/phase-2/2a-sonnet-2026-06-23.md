# Phase 2a — Sonnet Code Review
**Date:** 2026-06-23  
**Sub-phase:** 2a — Tasks REST API  
**Reviewer:** claude-sonnet-4-6

## Findings

### [CONFIRMED] 1. Invalid `dueDate` string causes HTTP 500 instead of 400
**File:** `server/src/routes/tasks.ts` lines 65 (POST) and 103 (PATCH)  
**Severity:** High  
**Scenario:** `POST /api/tasks` with `{ dueDate: "not-a-date" }` passes the `typeof dueDate === 'string'` guard. `new Date("not-a-date")` produces an Invalid Date object. Prisma internally calls `.toISOString()` on it, which throws `RangeError: Invalid time value`. The catch block routes to `next(err)`, errorHandler returns HTTP 500 — caller gets no indication the date was invalid.  
**Fix:** Guard with `isNaN(new Date(dueDate).getTime())` → return 400.

### [CONFIRMED] 2. Empty PATCH body calls `prisma.task.update` with `data: {}` and returns HTTP 200
**File:** `server/src/routes/tasks.ts` line 99–106  
**Severity:** Medium  
**Scenario:** `PATCH /api/tasks/:id` with `{}` (or body containing only unknown keys) — none of the if-branches fire, `data` stays `{}`. Prisma v6 + SQLite performs a no-op (no UPDATE emitted) and returns the existing record. Caller receives HTTP 200 with unchanged data and no signal that nothing happened.  
**Fix:** After building `data`, check `Object.keys(data).length === 0` and return 400.

### [PLAUSIBLE] 3. `dotenv.config()` executes after `PrismaClient` is constructed
**File:** `server/src/index.ts` line 2  
**Severity:** Low (currently mitigated by Prisma's own internal .env loader)  
**Scenario:** In compiled CommonJS, `require('./app')` (line 2) fires before `dotenv.config()` (line 4). The import chain app.ts → tasks.ts → lib/prisma.ts → `new PrismaClient()` all run before .env is loaded. Prisma's internal .env loader saves `DATABASE_URL` under the current setup, but any future module-load-time env read outside Prisma would see an empty environment.  
**Fix:** Move `dotenv.config()` into a dedicated `config.ts` module imported as the very first import in `index.ts`.

### [REFUTED] 4. `!priority` falsy check on line 51
Priority enum values are `"LOW"`, `"MEDIUM"`, `"HIGH"` — all truthy strings. `!priority` can never incorrectly reject a valid enum value. Redundant but not a bug.

## Verdict: 3 findings to fix (2 confirmed, 1 plausible structural risk)
