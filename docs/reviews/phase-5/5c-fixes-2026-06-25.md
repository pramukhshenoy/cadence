# Phase 5c Fixes — 2026-06-25

## Sonnet findings applied (4 total)

1. **`getActiveFutureBlocks()` excludes already-started morning blocks** (`use-sleep-reschedule.ts`, `server/src/routes/sleep.ts`)
   - Changed `POST /api/sleep` to return `blocksToDelete` as `{id, deviceCalendarEventId}[]` (was `string[]`). Client now reads `deviceCalendarEventId` directly from the plan — no `getActiveFutureBlocks()` call needed, and blocks that started before `now` are no longer missed.

2. **`getActiveFutureBlocks()` called before `postSleepRecord` — failure suppresses sleep POST** (`use-sleep-reschedule.ts`)
   - Removed `getActiveFutureBlocks()` entirely (see fix 1). `postSleepRecord` is now called first, so the sleep record is persisted even if subsequent calendar operations fail.

3. **Server test assertions updated for new `blocksToDelete` shape** (`server/src/__tests__/sleep.test.ts`)
   - Updated two assertions to map `(b) => b.id` / `(b) => b.deviceCalendarEventId` before asserting, matching the new response shape.

4. **Guard reset on error removed** (`use-sleep-reschedule.ts`)
   - Removed `queryClient.setQueryData(RESCHEDULE_ATTEMPTED_KEY, undefined)` from the catch block. Guard stays set on error to prevent retry storms when the backend is temporarily down.

## Opus findings applied (2 total)

1. **`FOCUS_WEEK_SUMMARY_KEY` not invalidated when blocks archived but no new blocks placed** (`use-sleep-reschedule.ts`)
   - Added `queryClient.invalidateQueries({ queryKey: FOCUS_WEEK_SUMMARY_KEY })` immediately after `archiveFocusBlocks` completes, before the early-exit on `newBlocks.length === 0`. This ensures the dashboard focus widget reflects archived blocks even when all slots are full (droppedCount > 0) or all `writeFocusBlock` calls fail.

2. **Map + second-pass redundancy in sleep route** (`server/src/routes/sleep.ts`)
   - Replaced `Map(morningBlocks.map(...))` + `reschedule.blocksToDelete.map(id => ...)` with a single direct `morningBlocks.map(b => ({id: b.id, deviceCalendarEventId: b.deviceCalendarEventId}))`. Valid because `computeReschedule` always sets `blocksToDelete = morningBlocks.map(b => b.id)` (no filtering).
