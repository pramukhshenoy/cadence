## Findings (ranked most severe first)

1. **CRITICAL** `app/src/hooks/use-sleep-sync.ts:100` — `useSleepSummary`'s `queryFn: () => null` executes on first mount when cache is empty, setting `SLEEP_SUMMARY_KEY = null`; Effect 1's `existing !== undefined` guard then sees `null`, exits immediately, Health Connect is never read.

2. **CRITICAL** `app/src/hooks/use-sleep-sync.ts:51` — When real sleep data exists, Effect 1 writes only `RAW_SLEEP_KEY`; relies on Effect 2 to write summary. But Effect 2 runs before Effect 1's async completes (finds RAW_SLEEP_KEY=undefined, exits), and never re-runs after Effect 1 finishes. SLEEP_SUMMARY_KEY is never populated.

3. **HIGH** `app/src/hooks/use-sleep-sync.ts:63` — Error catch sets `SLEEP_SUMMARY_KEY=null`, which permanently blocks all retries (same null !== undefined guard). Transient error = permanent lockout.

4. **HIGH** `app/src/hooks/use-sleep-sync.ts:39` — Guard uses `SLEEP_SUMMARY_KEY` but should use `RAW_SLEEP_KEY`. SLEEP_SUMMARY_KEY is poisoned by useSleepSummary's queryFn and the error handler; RAW_SLEEP_KEY is only written on a real successful HC read.

5. **MEDIUM** `app/src/hooks/use-sleep-sync.ts:90` — `rescheduledCount` hardcoded to 0 — any threshold change via settings overwrites the actual count that phases 5b/5c will write.

6. **LOW** `app/src/hooks/use-sleep-sync.ts:76` — Alias locals `thresholdHours`/`goodHours` in synchronous Effect 2 are unnecessary; TypeScript narrows after the undefined guard on line 74.

7. **LOW** `server/src/routes/settings.ts:87` — Gratuitous DB read when both threshold fields are provided in one PATCH (current values fetched but neither ?? fallback fires).

8. **LOW** `app/src/hooks/use-sleep-sync.ts:100` — Fix: `useSleepSummary` should use `skipToken` or `enabled: false` so TanStack Query never executes queryFn and never primes the cache.
