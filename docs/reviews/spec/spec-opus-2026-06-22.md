## HIGH severity

### H1. Calendar mechanism contradicts project memory
**Files:** `architecture.md`, `features.md` §5, `phases.md` Phase 4, `setup.md` vs. `memory/project_ai_assistant.md`
Spec describes expo-calendar (on-device, on-app-open). Memory still records Google Calendar API + nightly node-cron. These are mutually exclusive. expo-calendar cannot run "nightly" — there is no reliable always-on background execution in React Native for arbitrary cron-like work.
**Fix:** Memory file is stale. Spec is correct. Update memory to remove Google Calendar API / nightly cron references.

### H2. Sleep quality thresholds have overlap and gap
**Files:** `features.md` §6, `data-model.md` `Settings`
Good ≥ 7h, Fair 6–7h, Poor < 6.5h — these overlap between 6.0–6.5h and leave the configurable threshold disconnected from Fair/Good boundaries. "Adequate deep + REM" is unquantified.
**Fix:** Define as: Poor `< sleepThresholdHours`, Fair `sleepThresholdHours ≤ total < goodThresholdHours`, Good `≥ goodThresholdHours`. Add `goodThresholdHours` to Settings. Specify inclusive/exclusive bounds.

### H3. Expo Go cannot run Health Connect
**Files:** `architecture.md` (Deployment), `dev-workflow.md` (Testing Setup)
`react-native-health-connect` is a native module not included in Expo Go. Health Connect requires an **EAS development build** (`expo-dev-client`). Also, Health Connect cannot be meaningfully tested on the Android emulator — a physical device with a real sleep tracker is required.
**Fix:** Specify EAS dev-client builds for Phases 4–5. Update dev-workflow.md. Remove implication that `expo start → press 'a'` works for Health Connect.

### H4. FocusBlock model lacks idempotency — duplicate/orphaned blocks on re-sync
**Files:** `data-model.md` `FocusBlock`, `features.md` §5 re-scheduling flow
No `@unique` on `deviceCalendarEventId`. No status/lifecycle field. The "delete stale future blocks" step has no reliable predicate. Delete-from-calendar and update-DB are not transactional — a network failure between steps leaves device and server diverged with no reconciliation.
**Fix:** Add `@unique` on `deviceCalendarEventId`. Add `status` enum (`ACTIVE`/`DELETED`). Store a stable app-generated UUID in the calendar event notes field for reliable identification. Define partial-failure recovery.

### H5. No timezone handling anywhere
**Files:** `data-model.md` (all DateTime fields), `features.md` §5 and §6, `Settings` hour fields
The entire app is date/time-centric, but no timezone strategy exists. SQLite stores DateTime as UTC. `workdayStartHour`, `morningCutoffHour` are bare integers with no timezone context. "Today's tasks," "last night's sleep," "skips weekends," and streak logic all depend on the device's local day boundary while the backend computes schedules. `HabitCompletion.date` stored as DateTime in UTC will break streak logic at timezone boundaries.
**Fix:** Pass device IANA timezone with each sync/sleep/calendar request. Store habit completions as a local date string (`YYYY-MM-DD`), not DateTime. Document timezone contract in data-model.md.

### H6. No authentication on the backend
**Files:** `architecture.md`, `setup.md`, `phases.md`
Backend exposes full CRUD, chat (using the Anthropic key), and calendar sync with no authentication. Anyone on the same Wi-Fi can burn Claude credits or read/modify all data.
**Fix:** Add a shared bearer token between app and backend, stored in Expo SecureStore, validated by Express middleware. Add to Phase 1.

## MEDIUM severity

### M1. Streaming responses conflicts with REST and TanStack Query
**Files:** `architecture.md` (REST arrow), `features.md` §4, `phases.md` Phase 3
Streaming Claude responses requires SSE or chunked transfer. React Native `fetch` does not support streaming bodies on Android without additional libraries. TanStack Query is request/response oriented and not suited for token streaming.
**Fix:** Specify SSE as the streaming transport. Note that chat streaming needs a custom hook, not TanStack Query. Add `react-native-sse` or equivalent to tech stack.

### M2. ChatMessage has no session/conversation grouping
**Files:** `data-model.md` `ChatMessage`
Flat history with no `conversationId`. No start-new-chat capability. No context-window growth policy — full history + tasks injected into every prompt will eventually hit model limits.
**Fix:** Add `Conversation` model or at minimum `conversationId` on `ChatMessage`. Define a context policy (last N messages or token budget).

### M3. expo-calendar target calendar is ambiguous
**Files:** `features.md` §5, `setup.md`
Android devices have multiple calendars (Google account, local, work profile). expo-calendar requires a specific `calendarId` for reads and writes. "Read all events" and "write focus blocks" need to specify which calendars. A read-only or wrong calendar fails silently.
**Fix:** Add a calendar selection step to onboarding. Persist the target `calendarId` in Settings. Option: create a dedicated local "AI Focus Blocks" calendar.

### M4. Health Connect data freshness not handled
**Files:** `features.md` §6, `phases.md` Phase 5
Sleep trackers sync on their own schedule — data may not exist at app-open time. Multiple sleep sessions per night (naps, fragmented) are unhandled. `SleepRecord` has no unique constraint on `date`, so repeated reads create duplicate rows.
**Fix:** Define behavior when data is missing (skip reschedule, show "no data"). Define aggregation for multiple sessions. Add unique constraint on `date` in `SleepRecord`.

### M5. Weekend skip not configurable; capacity overrun undefined
**Files:** `features.md` §5 scheduling rules, `Settings`
"Skips weekends by default" implies a setting that doesn't exist. If `focusHoursPerWeek` exceeds available gaps, the algorithm's behavior (partial fill? error?) is undefined.
**Fix:** Add `includeWeekends` to Settings. Define under-capacity behavior (fill what's available, show shortfall on dashboard).

### M6. Sleep reschedule has no fallback for full afternoon
**Files:** `features.md` §6 step 3
If the afternoon is fully booked, no fallback is defined (next day? drop? evening?). Multiple displaced morning blocks competing for limited afternoon slots is unhandled.
**Fix:** Define fallback ordering: afternoon today → tomorrow morning → drop with dashboard notice. Handle multiple displaced blocks in sequence.

### M7. Model ID not verified; mixed-provider handling undefined
**Files:** `data-model.md` `Settings.preferredModel`, `phases.md` Phase 3
`"claude-sonnet-4-6"` is hardcoded and must exactly match the Anthropic model catalog. GPT-4o is listed in the UI but behavior when no `OPENAI_API_KEY` is set is undefined.
**Fix:** Verify model ID against current Anthropic model list. Hide GPT-4o option if `OPENAI_API_KEY` is absent.

### M8. Onboarding/server URL config deferred to Phase 6
**Files:** `phases.md` Phase 6 vs. Phases 1–5
Server URL is configured in onboarding (Phase 6) but needed from Phase 1. Calendar and Health Connect permissions are in Phase 6 onboarding but used in Phases 4–5.
**Fix:** Add a minimal server URL configuration (hardcoded default or simple input) to Phase 1. Move permission request flows to the phases where they're first needed.
